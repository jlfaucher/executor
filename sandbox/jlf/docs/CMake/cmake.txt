CMAKE_BUILD_TYPE

For single-config generators (e.g. Unix Makefiles, Ninja, NMake Makefiles) CMAKE_BUILD_TYPE is unset.
until:
- you pass -DCMAKE_BUILD_TYPE=..., or
- you set it inside your CMakeLists.txt.
So the initial value is literally empty.

For multi-config generators (e.g. Visual Studio, Xcode, Ninja Multi-Config)
These generators do not use CMAKE_BUILD_TYPE at all.
Instead they support multiple configurations simultaneously: Debug, Release, RelWithDebInfo, MinSizeRel.
The real configurations are stored in CMAKE_CONFIGURATION_TYPES

https://cmake.org/cmake/help/latest/guide/using-dependencies/index.html#guide:Using%20Dependencies%20Guide
Projects will frequently depend on other projects, assets, and artifacts.
CMake provides a number of ways to incorporate such things into the build.
The FetchContent module provides functionality to download content (typically
sources, but can be anything) and add it to the main project if the dependency
also uses CMake. The dependency's sources will be built along with the rest of
the project, just as though the sources were part of the project's own sources.


https://cmake.org/cmake/help/latest/module/FetchContent.html
This module provides commands to populate content at configure time or as part
of the calling script.

Example
https://github.com/cppalliance/corosio
    include(FetchContent)

    FetchContent_Declare(capy
        GIT_REPOSITORY https://github.com/cppalliance/capy.git
        GIT_TAG develop
        GIT_SHALLOW TRUE)
    FetchContent_Declare(corosio
        GIT_REPOSITORY https://github.com/cppalliance/corosio.git
        GIT_TAG develop
        GIT_SHALLOW TRUE)

    FetchContent_MakeAvailable(capy corosio)

    target_link_libraries(my_app Boost::corosio)
